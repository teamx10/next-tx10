# Lessons Learned

> Process and technical lessons.

- **2026-04-01 (#7):** Moving a provider (ThemeProviderWrapper) from the root layout to a nested locale layout creates silent breakage for sibling routes — `not-found.tsx`, `(legal)/privacy`, and `(legal)/terms` lost MUI theme context because they live outside `[locale]/`. When restructuring the layout hierarchy, audit ALL existing route paths to ensure routes outside the new nested layout still receive required providers.

- **2026-04-01 (#7):** next-intl v4 requires 4 config files in `lib/i18n/` (config.ts, routing.ts, request.ts, navigation.ts) vs v3's simpler inline config pattern. When upgrading from a planned library version to a newer major version, flag the API differences in `[IMPL]` comments and verify affected file counts match the plan — the v3→v4 migration added 2 unplanned files (routing.ts, navigation.ts) not listed in Files Touched.

- **2026-04-01 (#9):** When building a multi-section landing page, splitting into server vs client components by actual need (only SpeedComparison, HeroSection, CTASection needed 'use client') keeps the JS bundle lean. Server components with `getTranslations` avoid shipping translation runtime to the client.

- **2026-04-01 (#9):** Creating stub constants (services.ts, case-studies.ts) with clear "will be superseded by task #8" comments allows parallel task execution without blocking. The interface + data shape was defined upfront, so the landing page components work against a stable contract even before real content data lands.

- **2026-04-01 (#8):** Plan code snippets showed `interface ServiceFeature` (non-exported) in service.ts and `interface NavItem` (non-exported) in navigation.ts. Implementation copied verbatim, causing audit blockers when downstream consumers needed the types. When writing plan snippets, verify export visibility against downstream usage — not just the defining file's needs.

- **2026-04-01 (#8):** Ukrainian grammar error in services.ts (`Працюючий мультиагентний систем` instead of `Працююча мультиагентна система`) was caught only during audit code review. Non-English bilingual content should be grammar-checked before the first impl commit, especially for adjective/noun agreement in inflected languages.
- **2026-04-01 (#10):** For bilingual page content, resolving translations in the server page component and passing them as string props to content components (rather than using hooks in client components) keeps the client bundle lean and avoids hydration mismatches. When data already contains locale-keyed fields (`Record<Locale, string>`), access directly via `data.field[locale]` without translation runtime.
- **2026-04-01 (#11):** Passing `options` as an object literal to a custom hook's `useEffect` dependency array creates a new reference each render, potentially causing unnecessary re-runs. For one-shot observers that disconnect after triggering, the impact is minimal, but for reusable hooks the options should be memoized or individual values should be spread into the dependency array.
- **2026-04-01 (#11):** When a plan calls for "related services" in case study pages, defining the actual relationship (e.g., via a `relatedServiceSlugs` field in the data model) upfront avoids shipping a hardcoded `.slice(0, 2)` that shows the same 2 services for every case. For MVP this is acceptable, but the data model should be extended before adding more case studies.
- **2026-04-01 (#12):** When a polling `useEffect` (setInterval) checks for a global script to load, it should clear the interval once the condition is met (isAvailable = true), not poll indefinitely. The current useCalendly hook polls every 500ms even after Calendly loads — harmless but wasteful. Add a `return () => clearInterval(id)` guard inside the condition branch.
- **2026-04-01 (#12):** AboutPageContent hardcodes English strings for approach principles and values while only section headers are i18n'd. For MVP with a single-language body this works, but for true bilingual support all content strings should live in dictionaries. When scoping i18n tasks, explicitly state whether body content vs. only headers need translation.
- **2026-04-02 (#13):** When migrating CTA buttons site-wide to a centralized CTAButton component, the implementation cleanly removed unused imports (Button, Link, ROUTES) from 6 files. Standardizing UI patterns in a single final task avoids incremental drift — doing it as the last step catches all pages created by earlier tasks.
- **2026-04-02 (#13):** Stale `.next/dev/types/validator.ts` files reference old routes (products/, api/stripe/) that were removed in prior tasks, causing false tsc failures. When `npx tsc --noEmit` reports errors only in `.next/` generated files, the build passing is the authoritative check. Consider adding `.next/` to tsconfig exclude or cleaning the cache between major route restructures.
- **2026-04-02 (#24):** When an existing utility (`generateSEOMetadata`) already handles canonical, hreflang, OG, and Twitter metadata correctly, the most efficient SEO task is wiring it to all pages via a centralized dictionary namespace — 14 files touched but only ~20 lines of logic per page. Formalizing the task after implementation (spec-after-code) still provides value for audit and knowledge capture.
- **2026-04-02 (#24):** Referencing a not-yet-created asset (og-default.png 1200x630) in code is acceptable if documented with rollback plan, but should be tracked as a follow-up task to prevent indefinite 404s on social sharing previews.
